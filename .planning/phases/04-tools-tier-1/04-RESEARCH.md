## Summary

Phase 4 is primarily a **port-and-enhance** phase rather than a build-from-scratch phase. The calculation backbone is already complete: `chart.ts` (Placidus houses, 136 lines), `aspects.ts` (5 major aspects, 137 lines), `calculations.ts` (life path/destiny/soul/personality, 129 lines), `compatibility.ts` (12x12 matrix, 95 lines), and `prompts/interpretation.ts` (GEM 12 v6.0, 275 lines). All constants are typed and Hebrew-ready in `astrology.ts` and `numerology.ts` types.

The largest single piece of work is decomposing `BirthChart.jsx` (922 lines) into 4 typed SVG sub-components. The SVG geometry is fully defined: viewBox 500x500, center (250,250), planet positions calculated as `x = 250 + radius * cos((longitude - 90) * PI/180)`, ascendant as horizontal gold line, MC as vertical silver line to top. All 12 zodiac emoji/colors are already in `ZODIAC_SIGNS` constant.

The two "already built" tools (tarot and dream) need SubscriptionGuard verification only — both pages exist with full functionality. The personality analysis requires building from scratch because the BASE44 version is birth-date-based LLM analysis, not a Big Five questionnaire. The daily insights feature needs an API route that implements date-keyed cache logic against the `daily_insights` table (columns: `content`, `title`, `insight_date`, `data_sources`, `tarot`, `actionable_tip`).

**Primary recommendation:** Follow the plan order: natal chart SVG decomposition first (most complex), then interpretation API + calendar + forecast, then numerology enhancements, then daily insights cache, then tarot/dream verification, then personality Big Five, then SubscriptionGuard sweep.

### Pattern 1: SVG Coordinate Geometry (BirthChart)

**What:** All planet, house, and zodiac positions are calculated from ecliptic longitude (0-360°) mapped to SVG coordinates using a consistent formula.

**Key geometry (verified from BirthChart.jsx line 94-99):**
```typescript
// Source: github-source/src/components/BirthChart.jsx lines 94-99
// SVG viewBox="0 0 500 500", center = (250, 250)
// longitude=0° (Aries) maps to 9 o'clock (subtract 90° to rotate from East to top)
function getPlanetPosition(longitude: number, radius: number) {
  const angle = (longitude - 90) * (Math.PI / 180);
  return {
    x: 250 + radius * Math.cos(angle),
    y: 250 + radius * Math.sin(angle),
  };
}
```

**Radii to use (verified from BirthChart.jsx SVG elements):**
- Outer chart circle: r=240 (stroke only)
- Dashed reference circle: r=200 (zodiac separator)
- Planet orbit ring: r=160 (dashed)
- Planet positions: radius=170 passed to getPlanetPosition
- House number positions: radius=135
- Zodiac sign emoji positions: radius=220
- Aspect lines connect planet positions at radius=170

**What:** Show the reduction steps leading to the final numerology number. Required by D-13.

```typescript
// Example: life path from birthDate '1990-05-15'
// day=15 → 1+5=6
// month=5 → 5
// year=1990 → 1+9+9+0=19 → 1+9=10 → 1+0=1
// total=6+5+1=12 → 1+2=3
```

### Anti-Patterns to Avoid

- **Direct ZODIAC_SIGNS key access without type guard:** `ZODIAC_SIGNS[sign]` where `sign` is a plain `string` will break TypeScript strict — cast to `ZodiacSignKey` or use optional chaining.
- **`longitude` prop as `string` not `number`:** BirthChart.jsx passes `parseFloat(planet.longitude)` because its data came from a JSON with string numbers. New TypeScript types use `number` — no `parseFloat` needed.
- **SVG text in RTL container:** SVG text does NOT respect CSS `dir="rtl"`. Use explicit `textAnchor="middle"` or `textAnchor="end"` on SVG `<text>` elements. Do NOT rely on inherited RTL direction for SVG content.
- **Zod v4 breaking change:** `z.string().nonempty()` was removed in Zod v4. Use `z.string().min(1, 'חובה')` instead. The project uses zod v4.3.6.
- **date-fns v4 imports:** `import { format } from 'date-fns'` works fine, but locale imports changed. Use `import { he } from 'date-fns/locale/he'` (not `date-fns/locale`).
- **Recharts v3 Tooltip formatter:** Custom Tooltip component required for type-safe formatters — documented in STATE.md Decisions. Do NOT use `formatter` prop with inline functions.

## Common Pitfalls

### Pitfall 1: Type Mismatch Between `ChartData` and `BirthChart` Types

**What goes wrong:** `chart.ts` returns `ChartData` (with `houses: HouseData[]`). `types/astrology.ts` defines `BirthChart` (with `houses: House[]`). These are different types — `HouseData.sign` is `string`, `House.sign` is `ZodiacSign` (lowercase union). SVG components should accept `ChartData` (the service output), not `BirthChart` (the full aggregate type).

**Why it happens:** Two type systems exist — one from the service layer (chart.ts), one from the type file (astrology.ts). They were not unified during Phase 0.

**How to avoid:** Define SVG sub-component props to accept `ChartData` directly from `assembleChart()`. Do not try to transform into `BirthChart` until a unified adapter is needed. The ascendant in `ChartData` is a `number` (degrees), not `ZodiacSign` — compute the sign separately using `getSign(ascendant)`.

**Warning signs:** TypeScript error on `sign` property expecting `ZodiacSign` but receiving `string`.

### Pitfall 2: `PlanetPositions` Has No Actual Planet Data

**What goes wrong:** `chart.ts` returns `ChartData` with houses and ascendant, but planet longitudes are NOT computed — they are passed IN via `assembleChart(date, lat, lon, planets)`. The `planets` parameter must be provided by the calling API route. Phase 4 does not use a real ephemeris (that is Phase 6), so planet positions must be approximated.

**Why it happens:** `chart.ts` only computes houses (Placidus). Planet positions require an ephemeris library (deferred to Phase 6).

**How to avoid:** For Phase 4, the astrology API route must either (a) use simplified mean-longitude calculations for the 10 planets, or (b) use the BASE44-style approach: pass the birth date/time to `invokeLLM` and ask it to compute approximate planet positions in JSON format before feeding to `buildInterpretationPrompt`. Option (b) is less accurate but avoids blocking on ephemeris. Option (a) requires the simplified VSOP87 approximations already partially present in `solar-return.ts` (Sun calculation via Julian Date).

**Warning signs:** `assembleChart` is called but `planets` argument is empty `{}` — all charts show no planets.

**Recommended resolution (HIGH confidence):** Use a hybrid: compute Sun position via `solar-return.ts` Julian Date logic, use LLM to estimate remaining planet positions from birth date context. Flag clearly in the UI that positions are approximate until Phase 6.

### Pitfall 3: Daily Insights `insight_date` Timezone Drift

**What goes wrong:** Using `new Date().toISOString().split('T')[0]` returns UTC date, but user may be in a timezone where local date differs from UTC. A user at 11pm IL time would see "tomorrow's" insight date.

**Why it happens:** UTC vs local timezone mismatch in date strings.

**How to avoid:** Use the same pattern as DailyInsightCard in Phase 3 (documented in STATE.md): use UTC month/day for zodiac calculations. For daily insights caching, the date key should be determined server-side with explicit timezone awareness. Use the user's birth timezone (stored in profile) or Israel timezone as default.

**Warning signs:** User gets "no insight yet" in the evening even though one was generated earlier.

### Pitfall 4: SVG Text Rendering RTL Issues

**What goes wrong:** Placing Hebrew text inside SVG elements that are inside a `dir="rtl"` container causes bidirectional text rendering problems. The house numbers (1-12) might render in unexpected positions.

**Why it happens:** SVG text does not inherit CSS `direction: rtl`. SVG text positioning is always mathematical (absolute coordinates).

**How to avoid:** All SVG `<text>` elements use `textAnchor="middle"` and coordinate-based positioning. Do NOT use CSS text alignment inside SVG. House numbers are Latin numerals (1-12), not Hebrew — no RTL issue. If using Hebrew text in SVG tooltips, use shadcn Tooltip (React layer) not SVG `<title>` elements.

### Pitfall 5: SubscriptionGuard Scope in Tarot/Dream Pages

**What goes wrong:** The tarot page has `<SubscriptionGuard feature="analyses">` wrapping only the form content, not the page shell. The dream page has NO `<SubscriptionGuard>` at all — it was not added during original construction.

**Why it happens:** Dream page was built independently without the Guard wrapping.

**How to avoid:** For D-14 and D-15 (verify only tasks): check that `<SubscriptionGuard feature="analyses">` wraps the form submission button in tarot (confirmed at line 127) but verify dream page has it added. The dream page currently has no SubscriptionGuard import or usage — this is the verification failure.

**Warning signs:** Dream page allows unlimited submissions without plan enforcement.

### Pitfall 6: Big Five Questions — Reverse Scoring Logic

**What goes wrong:** Forgetting to reverse-score specific items. A question like "I am easily distracted" should score inverted (5 = low Conscientiousness). If not reversed, the dimension score is inflated/deflated.

**Why it happens:** Standard Big Five questionnaires always include reverse-scored items to prevent acquiescence bias.

**How to avoid:** Mark reverse-scored items in `big-five-questions.ts` constants with `isReversed: true`. Apply `effectiveScore = isReversed ? (6 - rawScore) : rawScore` before summing. Add a unit test verifying that a known all-1s response on a reverse-scored dimension gives maximum effective score.

### Pitfall 7: `invokeLLM` Token Cost for Daily Insights

**What goes wrong:** Daily insights call `invokeLLM` for every new day per user. With many users, this becomes expensive. Additionally, the generation must include a tarot card selection, numerology day number, and zodiac forecast — requiring careful prompt construction to avoid extra API calls.

**Why it happens:** D-05 explicitly states "costs tokens per user per day".

**How to avoid:** Cache in the DB immediately after generation (D-07). Use a single `invokeLLM` call with all three modules combined in the prompt, rather than three separate calls. Keep `maxTokens` around 1200 to cover all sections without runaway responses.

### Sub-Number Breakdown Display

```typescript
// Source: logic derived from calculations.ts reduceToSingleDigit
// Returns array of reduction steps for display
function getReductionSteps(num: number): number[] {
  const steps: number[] = [num];
  let current = num;
  while (current > 9 && current !== 11 && current !== 22 && current !== 33) {
    current = String(current).split('').reduce((sum, d) => sum + parseInt(d, 10), 0);
    steps.push(current);
    if (current === 11 || current === 22 || current === 33) break;
  }
  return steps;
}
// Example: getReductionSteps(29) → [29, 11]  (master number, stops)
// Example: getReductionSteps(28) → [28, 10, 1]
```

## Open Questions

1. **Planet position calculation for Phase 4 natal chart**
   - What we know: `assembleChart()` requires `PlanetPositions` input. No ephemeris is available until Phase 6. The Sun position can be computed from `solar-return.ts` Julian Date. Other planets need approximation.
   - What's unclear: How approximate is acceptable for Phase 4? The AI interpretation (ASTR-02) will reference specific placements — if they are wrong, interpretations are meaningless.
   - Recommendation: Use a two-step approach in the astrology API route. Step 1: Call `invokeLLM` with birth date/time/place to ask for approximate planet positions in JSON (zodiac signs + approximate houses) — this is what BASE44 originally did. Step 2: Feed those positions to `buildInterpretationPrompt`. The SVG will show approximate positions. Flag in UI: "מיקומים מקורבים — Phase 6 יוסיף חישובים מדויקים." This unblocks Phase 4 without real ephemeris.

2. **Astro calendar event generation strategy**
   - What we know: AstroCalendar.jsx uses `generateSampleEvents` (hardcoded). The decision is LLM-generated events for each month.
   - What's unclear: Should events be generated fresh per month via LLM (one call per calendar navigation), or pre-seeded with static astronomical data for the next 12 months?
   - Recommendation: LLM-generated per month, cached in DB with `(month, year)` key. Simple and consistent with the project's AI-first approach.

3. **Tarot card for daily insights — random selection**
   - What we know: D-05 specifies "random tarot card" as one of the three inputs. The tarot cards are in the `tarot_cards` table.
   - What's unclear: Should the daily insights API select the tarot card randomly from DB, or should it call the tarot service?
   - Recommendation: SELECT random tarot card directly via `ORDER BY RANDOM() LIMIT 1` in Supabase query — avoid calling the full tarot service to keep daily insights generation to a single LLM call.
